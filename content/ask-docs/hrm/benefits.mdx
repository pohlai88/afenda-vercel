## Overview

**Benefits** covers plan catalogs, employee enrollments, and the time windows when changes are allowed. Enrollment mutations are **Server Actions** — there is no shadow REST API — so every commit is attributable.

## Before you start

- Maintain the plan catalog and tiers before opening enrollment to employees.
- Configure **open enrollment** windows with explicit start and end dates.
- Life-event changes require evidence when policy demands it — prepare review queues for HR.

## Managing enrollments

<Steps>
  <Step>
    Navigate to **HRM → Benefits** and review the active plan catalog.
  </Step>
  <Step>
    Open or schedule an **open enrollment** window when self-service changes should be allowed.
  </Step>
  <Step>
    Process employee enrollments, waivers, or life-event requests with evidence attached.
  </Step>
  <Step>
    Confirm deduction previews in **payroll** reflect enrolled benefits where configured.
  </Step>
</Steps>

## Plans & tiers

Plans describe coverage tiers, currencies, and eligibility notes. Administrators maintain the catalog; employees see read-only summaries in the portal where access is granted.

## Open enrollment

Open enrollment windows unlock self-service changes. Outside the window, employees request changes through HR — the portal surfaces the state honestly instead of failing silently.

## Life events

Qualifying life events (marriage, birth, loss of coverage) can unlock mid-year changes when policy allows. Evidence requirements stay explicit to support compliance reviews.

<Callout type="info" title="Portal parity">
  Open enrollment actions for employees live at [Employee portal benefits](/ask-docs/portals/benefits) when a window is active.
</Callout>

## Troubleshooting

<Accordions type="single">
  <Accordion title="Employee cannot enroll during open enrollment">
    Confirm the window is active, the plan is eligible for their population, and portal access is granted.
  </Accordion>
  <Accordion title="Payroll preview missing a deduction">
    Verify enrollment committed successfully and the benefit tier maps to a payroll component.
  </Accordion>
</Accordions>
