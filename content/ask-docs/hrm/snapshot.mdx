## Overview

The **HR snapshot** is a curated dashboard: it never invents numbers. Each card links into the owning HRM route so operators can move from signal to action without hunting URLs.

## Before you start

- Snapshot cards are read-only — mutations always happen in the linked execution module.
- Effective visibility follows ERP RBAC; cards for modules you cannot access may be hidden or show honest empty states.
- Refresh after completing Server Actions in downstream workflows — the snapshot shows last committed state, not in-flight drafts.

## Using the snapshot

<Steps>
  <Step>
    Open **HRM → HR snapshot** (`/o/{orgSlug}/dashboard/hrm` snapshot entry when enabled in your org).
  </Step>
  <Step>
    Scan pressure cards (payroll period state, compliance aging, headcount signals).
  </Step>
  <Step>
    Click a card deep link to open the owning module with context preserved.
  </Step>
  <Step>
    Return to the snapshot after resolving work to confirm pressure cleared.
  </Step>
</Steps>

## Signals vs actions

Cards may show pressure (e.g. aging compliance) but the snapshot itself does not mutate data. Mutations always happen inside the target module with ERP RBAC checks.

## Freshness

Snapshot queries are tuned for fast reads. If a downstream workflow is mid-flight, the snapshot shows the last committed state — refresh after Server Actions complete.

<Callout type="info" title="Read-only by design">
  The snapshot is an operational lens, not a second write path. If a number looks wrong, fix data in the source module — do not expect inline edits here.
</Callout>

## Troubleshooting

<Accordions type="single">
  <Accordion title="A card shows stale counts">
    Hard-refresh the page after downstream commits complete. Long-running payroll or compliance jobs may lag until the workflow finishes.
  </Accordion>
  <Accordion title="Expected card is missing">
    Your ERP role may lack search permission for that module, or the capability is disabled for the organisation.
  </Accordion>
</Accordions>
