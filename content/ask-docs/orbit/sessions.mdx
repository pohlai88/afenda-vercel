## Overview

**Orbit Sessions** (`/{locale}/o/{orgSlug}/dashboard/orbit/sessions`) shows **active and recent execution sessions** — focused work periods logged against planner items with worklog continuity.

Use Sessions when you need to answer: *who is executing what right now, and what time was spent?*

## Before you start

- You need an active org session and ERP permissions for the planner module.
- Sessions are first-class entities — not a hidden column on items.
- Starting a session emits `erp.planner.session.start`; stopping emits `erp.planner.session.stop`.

## What is a session?

A **planner session** is a bounded period of execution:

- **Started** with an explicit operator action
- **Stopped** when work pauses or completes
- Linked to one or more execution items for evidence and reporting

<Callout type="info" title="Evidence, not timesheets alone">
  Session history rolls into **Execution record** and **Activity stream** evidence groups on items — supporting audits and operator handoffs.
</Callout>

## Start and stop a session

<Steps>
  <Step>
    From [Queue](/ask-docs/orbit/queue) or item detail, click **Start session** (or use Sessions quick-create when no item is pre-selected).
  </Step>
  <Step>
    Work the item — comments, attachments, and lifecycle transitions accrue as operator evidence while the session runs.
  </Step>
  <Step>
    **Stop session** when pausing or finishing the focused block. Stopping prevents orphaned running sessions.
  </Step>
</Steps>

## Sessions surface

The Sessions feed lists:

- **Active** sessions — currently running
- **Recent** sessions — stopped within the retention window shown in the UI

Select a session to inspect linked items, duration evidence, and related activity in the detail pane.

## Worklog continuity

Session history rolls into execution evidence groups (**Execution record**, **Activity stream**) on items. This supports audits and handoffs — the next operator can see that work was in progress even if lifecycle state is still `active`.

## Troubleshooting

<Accordions type="single">
  <Accordion title="I cannot start a session">
    Confirm an execution item is selected or created first. Sessions attach to planner items, not bare signals awaiting triage.
  </Accordion>
  <Accordion title="A session still shows as active">
    Stop the session explicitly when pausing work. Only one active session pattern should run per operator policy.
  </Accordion>
  <Accordion title="Session time is missing from item evidence">
    Open the item detail pane and check **Execution record** — session stop must complete before history appears in evidence groups.
  </Accordion>
</Accordions>
