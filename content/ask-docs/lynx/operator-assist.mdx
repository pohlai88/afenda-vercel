## Overview

**Operator assist** (preview) lets operators ask cross-module questions in natural language. Lynx may call **governed read-only tools**, then stream a recommendation. **Nothing in this layer writes ERP data** — mutations remain Server Actions in feature modules.

Route: **`POST /api/erp/lynx/operator`** · Audit: **`erp.lynx.operator.recommend`** · Permission: **`lynx.workspace.predict`**

## Before you start

- ERP permission **`lynx.workspace.predict`** on the operator route (page entry still requires **`lynx.workspace.search`**).
- **`FLAG_LYNX_OPERATOR_ENABLED`** — when off, the Lynx page shows a policy notice instead of the form.
- **`FLAG_LYNX_OPERATOR_ORBIT_TOOLS_ENABLED`** for Orbit-specific tools (defaults on when operator is enabled).
- AI Gateway auth for embedding-backed `org_search_knowledge`.

<Callout type="warn" title="Read-only by design">
  Operator tools declare `access: "read"` in v1. Lynx does not create signals, edit contacts, or post payroll from this layer.
</Callout>

## How to use operator assist

<Steps>
  <Step>
    Open **Lynx** at `/{locale}/o/{orgSlug}/dashboard/lynx` and scroll to **Operator assist (preview)**.
  </Step>
  <Step>
    Enter a request — for example: *What is highest pressure in Orbit right now, and which blocked items need recovery first?*
  </Step>
  <Step>
    Click **Ask Lynx**. The response streams as produced; up to **five** tool round-trips run per request (`LYNX_OPERATOR_MAX_STEPS`).
  </Step>
  <Step>
    Verify critical facts in the ERP UI, then act via the relevant module (Orbit, Contacts, imports) using normal Server Actions.
  </Step>
</Steps>

## Governed tools

Tools are tenant-bound: `organizationId` always comes from your session — never from model-supplied arguments.

### Contacts and knowledge

| Tool id | Purpose |
| --- | --- |
| `org_contact_count` | Total contacts in the directory |
| `org_recent_contacts` | Recently created contacts |
| `org_recent_knowledge_chunks` | Recent chunk titles and excerpts |
| `org_search_knowledge` | Semantic search over knowledge chunks |

### Operations

| Tool id | Purpose |
| --- | --- |
| `org_active_import_jobs` | CSV import jobs still in progress |

### Orbit (when orbit tools flag is on)

| Tool id | Purpose |
| --- | --- |
| `org_orbit_high_pressure` | Highest-pressure items and signals |
| `org_orbit_triage_snapshot` | Pending triage items and incoming signals |
| `org_orbit_blocked_items` | Blocked items with duration and facts |
| `org_orbit_item_detail` | Full item snapshot by UUID |
| `org_orbit_signal_detail` | Full signal snapshot by UUID |

## Truth retrieval vs operator assist

| Use truth retrieval when… | Use operator assist when… |
| --- | --- |
| Answer should cite **written** policy chunks | You need **live** counts, lists, or Orbit state |
| Question is document-centric | Question spans modules or needs tabular facts |
| You are debugging chunk coverage | You are coordinating operations across systems |

## Troubleshooting

<Accordions type="single">
  <Accordion title="Operator section is dashed / disabled">
    Check `FLAG_LYNX_OPERATOR_ENABLED` and `lynx.workspace.predict`. The API returns 403 when policy blocks the feature.
  </Accordion>
  <Accordion title="Orbit tools never appear in answers">
    Confirm `FLAG_LYNX_OPERATOR_ORBIT_TOOLS_ENABLED` — without it, the registry omits Orbit tool ids.
  </Accordion>
  <Accordion title="Knowledge search tool errors">
    Gateway auth and chunk embeddings must be available — fix substrate first, then retry operator assist.
  </Accordion>
</Accordions>
