## Overview

Lynx is Afenda's **machine layer** for resolving truth, preparing work, and recommending controlled action. It is an ERP module (`lib/features/lynx`) with the same tenancy and audit expectations as HRM or Contacts. Inference providers (Vercel AI Gateway, OpenAI embeddings) are implementation details тАФ not separate product brands.

## Before you start

- Read [Lynx overview](/ask-docs/lynx) for surface inventory and permission keys.
- Distinguish **ERP Lynx** (`#features/lynx`, org session) from **Public Ask Lynx** (`lib/ask-docs/public-lynx*`, no tenant data).
- Review banned vocabulary before writing product copy or internal runbooks.

<Callout type="info" title="Public vs ERP Lynx">
  Ask Lynx on `/{locale}/ask-docs` uses public docs search only. ERP Lynx routes under `app/api/erp/lynx/*` require org session and ERP permissions (`lynx.workspace.search`, `lynx.workspace.predict`).
</Callout>

## Machine layer doctrine

```txt
Lynx is Afenda's machine layer for resolving truth, preparing work, and recommending controlled action.
```

### What Lynx is not

| Misconception | Reality |
| --- | --- |
| Generic AI assistant / chatbot | Governed **Lynx** with evidence and audit |
| Workflow automation | Durable runs belong in **execution** / Workflow DevKit |
| Replacement for Orbit | Lynx **reads** operational state; Orbit **owns** execution |
| Public docs chat | Separate route тАФ [Ask Lynx on docs](/ask-docs/lynx/ask-docs-chat) |

## Banned user-facing vocabulary

Do not use in product copy or navigation: **AI assistant**, **chatbot**, **copilot**, **AI mode**, **Thinking**, **Processing**, **Generating** (as status labels).

Prefer: **Lynx**, **The Machine**, **Resolving**, **listening**, **high confidence**, and `Dashboard.Lynx.*` section titles.

## Knowledge vs Lynx

| Module | Role |
| --- | --- |
| **`#features/knowledge`** | Substrate тАФ `knowledge_chunk`, embeddings, chunk CRUD, similarity search |
| **`#features/lynx`** | Product тАФ truth stream, operator tools, NLтЖТSQL demo, reserved briefs/intake |

Lynx imports Knowledge only through `#features/knowledge`. Adapters list documents; only the pipeline commits chunks. Retrieval is **source-blind** тАФ ranking uses embeddings and tenant scope.

## Four layers

| Layer | Code slug | Status in product |
| --- | --- | --- |
| Truth retrieval | `truth` | **Live** тАФ `/api/erp/lynx/truth-search` |
| Operating briefs | `briefs` | Contract + audit reserved тАФ no UI |
| Canonical intake | `structured` | NLтЖТSQL **demo** when flag enabled |
| Decision operator | `operator` | **Live (preview)** тАФ `/api/erp/lynx/operator` |

Canonical metadata: `lib/features/lynx/lynx.contract.ts` (`LYNX_LAYERS`, `LYNX_AUDIT_ACTIONS`).

## Truth response shape

Every truth UI surfaces four blocks:

1. **Answer** тАФ synthesis from retrieved passages only
2. **Evidence used** тАФ chunk titles, excerpts, cosine distance
3. **Limitations** тАФ coverage gaps
4. **Next safe action** тАФ operator guidance without bypassing policy

## HTTP and import boundaries

| Route family | Purpose |
| --- | --- |
| `app/api/erp/lynx/*` | Authenticated streaming (truth, operator) |
| `app/api/chat/*` | Public Ask Lynx тАФ **no** `#features/lynx` import |

Client islands use `#features/lynx/client` only (`LYNX_ERP_HTTP_ROUTES`, `parseLynxTruthMarkdown`).

## Troubleshooting

<Accordions type="single">
  <Accordion title="Which Lynx should I document or build against?">
    Tenant features use `#features/lynx` and `app/api/erp/lynx/*`. Public docs AI uses `lib/ask-docs/public-lynx*` and `app/api/chat/route.ts` тАФ never cross-import.
  </Accordion>
  <Accordion title="Why are briefs and intake in the contract but not in UI?">
    `erp.lynx.brief.generate` and `erp.lynx.intake.commit` are reserved audit strings for future layers тАФ do not document them as shipped product surfaces.
  </Accordion>
  <Accordion title="Operator tool called with wrong tenant">
    Impossible by design тАФ `organizationId` is injected from session in the tool registry, not from model arguments.
  </Accordion>
</Accordions>
